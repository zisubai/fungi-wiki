# 微生物百科到合成生物设计辅助平台需求规划

## 1. 项目愿景

项目按照以下路径逐步演进：

```text
微生物百科 → 功能菌数据库 → 菌种智能搜索 → AI 菌种推荐 → 合成生物设计辅助平台
```

核心目标不是一开始做成大而全的科研平台，而是先沉淀高质量菌种数据，再逐步增强搜索、推荐和设计辅助能力。平台最终应服务于农业、环保、食品、医药、工业发酵和合成生物学等场景，帮助用户从“查菌”升级到“选菌、配菌、设计方案”。

## 2. 产品定位

### 2.1 微生物百科

面向普通用户、学生、科研人员和产业用户，提供菌种基础知识、分类信息、功能说明、应用案例和文献来源。

主要价值：

- 降低用户理解微生物和功能菌的门槛。
- 建立可持续运营的内容入口。
- 为后续功能菌数据库积累标准化数据。

### 2.2 功能菌数据库

将百科内容升级为结构化、可检索、可计算的数据资产。

主要价值：

- 建立菌种、功能、环境条件、应用场景、文献证据之间的关系。
- 为智能搜索和 AI 推荐提供可信数据基础。
- 支持企业和科研用户进行菌种筛选、对比和数据分析。

### 2.3 菌种智能搜索

从“按名称搜索”升级为“按需求搜索”。用户可以输入自然语言需求，也可以使用结构化筛选条件查找菌种。

主要价值：

- 提升用户找菌效率。
- 支持功能、场景、环境条件、代谢产物、文献证据等多维检索。
- 为后续推荐系统积累搜索日志和用户意图数据。

### 2.4 AI 菌种推荐

基于用户需求、功能菌数据库、文献证据和规则评分，推荐候选菌种、组合菌群和验证思路。

主要价值：

- 将数据库能力转化为决策辅助能力。
- 给出推荐理由、适用条件、风险提示和证据来源。
- 支持农业、环保、发酵、食品和研发场景下的菌种初筛。

### 2.5 合成生物设计辅助平台

在数据、搜索和推荐能力成熟后，进一步支持底盘菌选择、代谢通路分析、基因模块查询、设计方案生成和实验验证辅助。

主要价值：

- 面向科研人员和企业研发团队提供专业设计工具。
- 支持从目标产物或功能目标反推底盘菌、通路和候选基因。
- 形成数据、AI、专家审核和实验反馈闭环。

## 3. 用户角色

### 3.1 普通用户

典型需求：

- 浏览微生物百科。
- 查询某个菌种的基本信息。
- 了解菌种用途、风险和应用案例。

### 3.2 科研用户

典型需求：

- 查询菌种分类、功能、培养条件和文献证据。
- 对比多个候选菌种。
- 查看功能验证方式和实验条件。
- 导出数据用于科研分析。

### 3.3 企业用户

典型需求：

- 按行业场景查找可用功能菌。
- 关注应用成熟度、安全性、产业化案例和成本。
- 寻找菌种组合、替代菌种和项目方案。

### 3.4 运营用户

典型需求：

- 录入、编辑和审核菌种数据。
- 管理功能标签、应用场景和文献证据。
- 维护专题内容和推荐内容。
- 处理用户反馈和纠错。

### 3.5 专家用户

典型需求：

- 审核高风险数据、推荐结果和设计方案。
- 标注文献证据强度。
- 维护功能分类体系和实验数据标准。

## 4. 阶段需求规划

## 4.1 第一阶段：微生物百科 MVP

### 阶段目标

建立可上线的内容型产品，完成菌种基础数据展示、搜索和后台维护。

### 核心功能

用户端：

- 首页展示热门菌种、功能分类和应用专题。
- 菌种百科列表。
- 菌种详情页。
- 基础关键词搜索。
- 按分类、功能、应用场景浏览。

运营端：

- 菌种词条新增、编辑、删除。
- 菌种分类管理。
- 功能标签管理。
- 应用场景管理。
- 文献链接维护。
- 内容发布和下架。

后端：

- 菌种基础信息 API。
- 分类和标签 API。
- 搜索 API。
- 运营管理 API。
- 用户权限基础能力。

### 数据范围

建议首期建设 500–2000 条高质量菌种数据，优先覆盖：

- 农业功能菌。
- 环保降解菌。
- 食品发酵菌。
- 工业产酶菌。
- 常见模式菌和底盘菌。

### 验收指标

- 可录入并发布菌种词条。
- 用户可搜索并查看菌种详情。
- 每个菌种至少包含名称、分类、功能、应用和来源。
- 后台可维护功能标签和应用场景。

## 4.2 第二阶段：功能菌数据库

### 阶段目标

将百科内容结构化，形成支持筛选、分析和推荐的数据资产。

### 核心功能

用户端：

- 高级筛选。
- 菌种对比。
- 功能菌专题库。
- 数据完整度展示。
- 文献证据展示。

运营端：

- 功能数据维护。
- 培养条件维护。
- 实验数据维护。
- 文献证据维护。
- 数据质量评分。
- 数据版本记录。

后端：

- 菌种主数据服务。
- 功能标签服务。
- 文献证据服务。
- 实验条件服务。
- 数据质量评分服务。

### 关键数据模型

菌种基础数据：

- 拉丁名。
- 中文名。
- 同物异名。
- 分类地位。
- 来源环境。
- 保藏编号。
- 生物安全等级。
- 是否模式菌。

功能数据：

- 功能类型。
- 功能描述。
- 功能强度。
- 验证方式。
- 适用场景。
- 适用环境。
- 证据来源。

实验条件：

- 培养基。
- 温度范围。
- pH 范围。
- 盐度范围。
- 氧需求。
- 培养时间。
- 产物或指标。

证据数据：

- 文献标题。
- DOI / PMID / 专利号。
- 数据来源。
- 实验结论。
- 证据等级。
- 审核状态。

### 验收指标

- 支持按功能、场景和环境条件组合筛选。
- 每条核心功能数据有关联证据。
- 具备数据审核状态和质量评分。
- 支持菌种对比。

## 4.3 第三阶段：菌种智能搜索

### 阶段目标

让用户可以用自然语言或复杂条件查找菌种。

### 核心功能

用户端：

- 自然语言搜索。
- 高级筛选搜索。
- 混合搜索结果。
- 搜索结果解释。
- 搜索结果收藏。
- 搜索历史。

运营端：

- 搜索词分析。
- 无结果查询分析。
- 热门需求统计。
- 搜索同义词管理。
- 标签召回规则配置。

后端：

- 全文搜索服务。
- 语义搜索服务。
- 搜索排序服务。
- 搜索日志服务。
- 同义词和术语标准化服务。

### 搜索能力演进

第一步：字段搜索。

```text
菌名、中文名、功能标签、应用场景、文献标题
```

第二步：全文搜索。

```text
菌种描述、功能说明、应用案例、文献摘要
```

第三步：语义搜索。

```text
用户输入“适合盐碱地促生的菌”可以匹配耐盐、促生、农业应用相关菌种
```

第四步：混合搜索。

```text
关键词召回 + 向量召回 + 条件过滤 + 规则排序
```

### 验收指标

- 支持自然语言需求搜索。
- 搜索结果能按相关性排序。
- 搜索结果展示匹配原因。
- 搜索日志可被运营端分析。

## 4.4 第四阶段：AI 菌种推荐

### 阶段目标

基于数据库和文献证据，为用户推荐候选菌种和初步应用方案。

### 核心功能

用户端：

- AI 推荐入口。
- 需求描述输入。
- 推荐菌种列表。
- 推荐理由展示。
- 证据来源展示。
- 风险提示。
- 替代菌种建议。
- 菌种组合建议。
- 推荐结果反馈。

运营端：

- 推荐结果审核。
- 推荐规则配置。
- 高风险关键词管理。
- 用户反馈分析。
- 推荐质量评估。

后端：

- 需求解析服务。
- 候选菌种召回服务。
- 推荐排序服务。
- RAG 证据聚合服务。
- 大模型生成服务。
- 反馈学习服务。

### 推荐流程

```text
用户需求
→ 意图解析
→ 功能标签识别
→ 应用场景识别
→ 环境条件识别
→ 候选菌种召回
→ 安全规则过滤
→ 证据强度评分
→ 推荐排序
→ AI 生成解释
→ 用户反馈回流
```

### 推荐评分示例

```text
总分 =
功能匹配度 * 0.35
+ 环境适配度 * 0.20
+ 文献证据强度 * 0.20
+ 应用成熟度 * 0.15
+ 安全合规性 * 0.10
```

### 安全原则

- AI 不直接凭空生成事实。
- 推荐必须关联数据库记录或文献证据。
- 高风险菌种、病原菌、耐药性相关内容需要安全提示。
- 涉及实验方案时需要标注“仅供科研参考”。
- 对合规风险较高的设计建议应进入专家审核。

### 验收指标

- 用户可输入需求并获得推荐列表。
- 推荐结果包含明确证据来源。
- 推荐结果包含风险提示和适用条件。
- 用户反馈可进入推荐质量分析。

## 4.5 第五阶段：合成生物设计辅助平台

### 阶段目标

面向科研和企业研发场景，支持从目标功能或目标产物出发进行设计辅助。

### 核心功能

用户端：

- 项目空间。
- 目标功能定义。
- 目标产物定义。
- 底盘菌推荐。
- 代谢通路查询。
- 关键酶和候选基因查询。
- 基因模块库。
- 初步设计方案生成。
- 风险和合规检查。
- 方案报告导出。

运营端 / 专家端：

- 底盘菌库维护。
- 通路数据维护。
- 基因模块维护。
- 设计规则维护。
- 专家审核工作台。
- 高风险方案拦截。

后端：

- 项目服务。
- 通路数据服务。
- 基因模块服务。
- 设计推理服务。
- 报告生成服务。
- 合规检查服务。

### 典型流程

```text
用户输入目标产物或功能
→ 系统识别目标类型
→ 查询相关通路和候选酶
→ 推荐底盘菌
→ 推荐候选基因来源
→ 生成初步设计思路
→ 输出风险点和验证建议
→ 专家审核或用户迭代
```

### 验收指标

- 支持底盘菌推荐。
- 支持通路和基因模块查询。
- 支持生成结构化设计报告。
- 高风险设计建议可被拦截或提示。

## 5. 数据采集规划

## 5.1 数据来源

公开数据库：

- NCBI Taxonomy。
- BacDive。
- LPSN。
- KEGG。
- UniProt。
- MetaCyc。
- GBIF。
- ATCC / DSMZ 等保藏信息。

文献和专利：

- PubMed。
- Crossref。
- Google Scholar。
- CNKI。
- 国家知识产权局专利数据。
- WIPO / Google Patents。

内部和运营数据：

- 专家录入数据。
- 实验室检测数据。
- 企业应用案例。
- 人工标注数据。
- 用户搜索和反馈数据。

## 5.2 数据采集流程

```text
数据源接入
→ 原始数据暂存
→ 字段清洗
→ 菌名标准化
→ 同物异名合并
→ 功能标签抽取
→ 应用场景标注
→ 文献证据关联
→ 数据质量评分
→ 人工审核
→ 正式入库
→ 建立搜索索引和向量索引
```

## 5.3 数据治理重点

菌名标准化：

- 处理拉丁名、中文名、旧名、新名、同物异名。
- 建立菌种唯一 ID。
- 保留原始来源名称，避免数据丢失。

功能标签标准化：

- 建立一级、二级、三级功能标签。
- 将自然语言描述映射到标准功能标签。
- 支持人工修正和标签合并。

证据质量管理：

- 区分数据库记录、综述、实验论文、专利、企业案例和用户反馈。
- 根据来源、实验类型和审核状态计算证据强度。
- 推荐结果优先使用高证据等级数据。

数据版本管理：

- 保存数据来源。
- 保存变更记录。
- 支持回滚和审计。
- 重要数据修改需要审核。

## 6. Go 语言开发方案

## 6.1 技术选型

后端语言：

- Go。

推荐原因：

- 编译部署简单。
- 并发能力强，适合搜索、采集和 API 服务。
- 性能稳定，适合中后台和数据服务。
- 生态成熟，适合快速构建工程化平台。

基础技术栈：

- Web 框架：Gin / Echo / Fiber。
- 数据库：PostgreSQL。
- 向量检索：pgvector，后期可扩展 Milvus。
- 搜索引擎：Meilisearch 起步，后期可升级 Elasticsearch / OpenSearch。
- 缓存：Redis。
- 任务队列：Asynq / RabbitMQ / Kafka。
- ORM：GORM / Ent。
- 配置：Viper。
- 日志：Zap / Zerolog。
- API 文档：OpenAPI / Swagger。
- 部署：Docker Compose 起步，后期 Kubernetes。

## 6.2 后端架构建议

早期建议采用模块化单体，不建议一开始拆成大量微服务。

推荐目录：

```text
cmd/server                应用启动入口
internal/user             用户与权限模块
internal/species          菌种数据模块
internal/taxonomy         分类模块
internal/function         功能标签模块
internal/literature       文献证据模块
internal/search           搜索模块
internal/recommend        AI 推荐模块
internal/admin            运营后台模块
internal/crawler          数据采集模块
internal/audit            审核模块
pkg/database              数据库连接
pkg/cache                 缓存封装
pkg/logger                日志封装
pkg/llm                   大模型调用封装
pkg/embedding             向量化封装
pkg/queue                 队列封装
api/openapi               API 文档
```

后期当业务复杂后，可平滑拆分为：

```text
api-gateway               API 网关
user-service              用户权限服务
species-service           菌种主数据服务
search-service            搜索服务
recommend-service         AI 推荐服务
crawler-service           数据采集服务
admin-service             运营后台服务
```

## 7. 前端、后端、运营端分层设计

当前项目目录较空，后续代码建议按照三端分层组织，避免用户端、后台和服务端混杂。

## 7.1 推荐代码结构

```text
apps/
  web/                       前端用户端
  admin/                     运营管理端
  api/                       Go 后端服务

packages/
  ui/                        前端通用 UI 组件
  shared/                    前后端共享类型或常量
  config/                    统一配置

docs/                        产品、技术和数据文档
scripts/                     数据脚本和运维脚本
deploy/                      部署配置
```

如果暂时不使用 monorepo，也可以先保持：

```text
frontend/                    前端用户端
admin/                       运营管理端
backend/                     Go 后端服务
docs/                        文档
```

当前最适合的建设方式是：

```text
第一步：微生物百科 + 功能菌基础库 + 后台录入审核
```
