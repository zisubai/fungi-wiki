# 前端、后端、运营端系统分层设计

## 1. 分层目标

当前项目应按用户端、运营端和后端服务端分层建设，避免页面、接口、数据采集和管理能力混在一起。分层后的目标是：

- 用户端专注浏览、搜索、推荐和数据展示。
- 运营端专注数据录入、审核、标注和质量管理。
- 后端专注 API、业务规则、数据存储、搜索、推荐和采集任务。

## 2. 推荐目录结构

建议采用以下结构：

```text
apps/
  web/                       用户前端
  admin/                     运营管理端
  api/                       Go 后端服务

packages/
  ui/                        前端通用组件
  shared/                    共享类型、常量、工具
  config/                    通用配置

docs/                        文档
scripts/                     数据采集、导入、运维脚本
deploy/                      Docker、Compose、K8s 配置
```

如果项目早期想保持简单，也可以使用：

```text
frontend/                    用户前端
admin/                       运营端
backend/                     Go 后端
docs/                        文档
scripts/                     脚本
deploy/                      部署配置
```

## 3. 用户前端

## 3.1 定位

面向普通用户、科研用户和企业用户，解决“查菌、找菌、理解菌、对比菌、获得推荐”的问题。

## 3.2 核心页面

- 首页：展示热门菌种、功能分类、应用场景和搜索入口。
- 微生物百科：展示菌种词条列表和分类浏览。
- 菌种详情：展示基础信息、功能、培养条件、文献证据和应用案例。
- 功能菌数据库：支持按功能、行业、条件、安全等级筛选。
- 智能搜索：支持关键词搜索、自然语言搜索和高级筛选。
- AI 推荐：支持输入需求并输出候选菌种、理由、证据和风险提示。
- 菌种对比：支持多菌种并列对比。
- 用户中心：收藏、搜索历史、推荐反馈和纠错记录。

## 3.3 模块划分

```text
src/main.tsx                 React 启动入口
src/App.tsx                  百科、搜索、详情和推荐页面编排
src/api.ts                   统一 API 请求与错误处理
src/types.ts                 用户端领域类型
src/App.test.tsx             搜索和推荐关键流程测试
src/features/species/        菌种列表与详情
src/features/search/         多条件搜索
src/features/recommend/      AI 推荐
```

## 4. 运营管理端

## 4.1 定位

面向运营、专家和管理员，解决“数据录入、数据审核、标签维护、文献证据、推荐质量和用户反馈”的问题。

## 4.2 核心页面

- 工作台：展示待审核数据、数据质量、搜索趋势和反馈统计。
- 菌种管理：新增、编辑、下架、合并同义菌名。
- 分类管理：维护分类体系和别名。
- 功能标签管理：维护功能标签层级和同义词。
- 文献管理：维护 DOI、PMID、标题、摘要、来源和证据等级。
- 实验条件管理：维护培养基、温度、pH、盐度和氧需求等数据。
- 应用案例管理：维护行业场景、应用结果和成熟度。
- 数据审核：支持新增、修改、删除、导入数据的审核流。
- 推荐审核：查看 AI 推荐结果，处理高风险或低置信度推荐。
- 用户反馈：处理纠错、无结果搜索、有用/无用反馈。

## 4.3 模块划分

```text
src/main.tsx                 React 启动入口
src/App.tsx                  应用壳、权限菜单和页面编排
src/api.ts                   统一请求、令牌和异常处理
src/types.ts                 运营端共享领域类型
src/components/              通用展示组件
src/features/auth/           登录与身份认证
src/features/analytics/      搜索分析和推荐质量
src/features/species/        菌种、培养条件和别名管理
src/features/tags/           功能标签管理
src/features/import/         批量导入和导入审核
src/features/evidence/       文献证据管理
src/features/audit/          发布审核
src/features/users/          账号与角色管理
```

新增运营页面时优先落入对应 `features/<domain>`，页面只通过 `api.ts` 访问后端，跨页面类型统一放入 `types.ts`，避免继续扩大应用入口文件。

## 5. Go 后端服务

## 5.1 定位

后端负责统一提供数据服务、搜索服务、推荐服务、采集任务和运营后台 API。

## 5.2 推荐工程结构

```text
apps/api/
  cmd/server/                HTTP 服务入口
  cmd/worker/                异步任务入口
  internal/user/             用户与权限
  internal/species/          菌种主数据
  internal/taxonomy/         分类体系
  internal/functiontag/      功能标签
  internal/literature/       文献证据
  internal/search/           搜索
  internal/recommend/        AI 推荐
  internal/crawler/          数据采集
  internal/audit/            审核流
  internal/admin/            运营端 API
  internal/feedback/         用户反馈
  pkg/database/              数据库连接
  pkg/cache/                 Redis 缓存
  pkg/logger/                日志
  pkg/queue/                 队列
  pkg/llm/                   大模型调用
  pkg/embedding/             向量化
  api/openapi/               OpenAPI 文档
```

## 5.3 API 分层

建议将接口分为三类：

用户端 API：

```text
GET  /api/species
GET  /api/species/{id}
GET  /api/tags
GET  /api/search
POST /api/recommendations
POST /api/feedback
```

运营端 API：

```text
POST   /api/admin/species
PUT    /api/admin/species/{id}
DELETE /api/admin/species/{id}
POST   /api/admin/literatures
POST   /api/admin/audit/{id}/approve
POST   /api/admin/audit/{id}/reject
GET    /api/admin/dashboard
```

内部任务 API 或任务处理：

```text
crawler.import_species
crawler.sync_literature
search.rebuild_index
embedding.build_species_vector
recommend.evaluate_quality
```

## 6. 数据流设计

## 6.1 内容录入流

```text
运营端录入菌种
→ 后端保存为草稿
→ 审核流生成审核记录
→ 专家或管理员审核
→ 发布正式数据
→ 更新搜索索引
→ 用户端可见
```

当前实现说明：菌种创建固定进入 `draft`；运营人员通过审核接口提交后进入 `pending_review`；审核通过进入 `published`，驳回返回 `draft`。用户端列表和详情均只允许访问已发布菌种。

## 6.2 数据采集流

```text
采集任务拉取外部数据
→ 原始数据入暂存表
→ 清洗和标准化
→ 匹配已有菌种
→ 生成待审核数据
→ 人工审核
→ 合并正式数据
→ 更新索引和向量
```

## 6.3 智能搜索流

```text
用户输入查询
→ 后端解析关键词和条件
→ 全文搜索召回
→ 向量搜索召回
→ 数据库条件过滤
→ 排序和去重
→ 返回结果和匹配原因
→ 记录搜索日志
```

## 6.4 AI 推荐流

```text
用户提交需求
→ 需求解析
→ 标签和场景识别
→ 候选菌种召回
→ 安全规则过滤
→ 证据聚合
→ 推荐评分
→ 大模型生成解释
→ 返回推荐结果
→ 用户反馈回流
```

## 7. 权限设计

建议角色：

- guest：未登录用户，只能浏览公开数据。
- user：普通用户，可收藏、反馈、使用基础推荐。
- researcher：科研用户，可查看更详细数据和导出部分数据。
- enterprise：企业用户，可使用项目空间和高级推荐。
- operator：运营用户，可录入和编辑数据。
- expert：专家用户，可审核高价值或高风险数据。
- admin：管理员，拥有系统配置和权限管理能力。

## 8. 开发顺序建议

第一步：

- 建立 `apps/web` 用户端。
- 建立 `apps/admin` 运营端。
- 建立 `apps/api` Go 后端。
- 建立菌种基础模型和 CRUD。

第二步：

- 打通用户端菌种列表和详情。
- 打通运营端菌种录入和编辑。
- 增加基础审核状态。

第三步：

- 增加功能标签、应用场景和文献证据。
- 增加基础搜索。
- 增加数据采集脚本。

第四步：

- 增加全文搜索和向量搜索。
- 增加 AI 推荐服务。
- 增加推荐反馈和质量评估。
